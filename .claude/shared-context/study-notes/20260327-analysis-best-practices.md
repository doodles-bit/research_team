# 분석 리포트 작성 Best Practices

> 2026-03-27 팀장 학습 → 사용자 검토 → 승인

---

## 최우선 원칙: 할루시네이션 금지

AI 에이전트 분석가의 최대 리스크는 그럴듯한 이야기를 지어내는 것이다.
데이터로 확인 안 된 건 말하지 않는다. "모르겠습니다"가 지어낸 답보다 낫다.

- 모든 수치에 출처(쿼리, CSV, URL, 테이블명) 필수
- 메모리 수치도 원본 데이터와 반드시 재확인
- 추정은 [Estimate] 태그 + 근거 명시
- 제언은 "데이터가 보여주는 사실 + 의사결정 필요 지점"까지만 — 솔루션 제시 금지

---

## 1. 분석가 마인드셋

### 가설 주도 분석
- 쿼리부터 짜지 말고 비즈니스 질문에서 출발
- "이 요청 뒤에 있는 진짜 질문은 무엇인가?" 먼저 파악

### "So What" 테스트
- 모든 섹션 끝에 "그래서 뭘 해야 하는데?" 필수
- 수치 나열 = 리포팅, 의미 해석 + 의사결정 지점 제시 = 분석

### 비즈니스 컨텍스트 우선
- "이 결과를 보고 어떤 의사결정을 하려는가?" 먼저 확인
- 기술적 완벽함보다 비즈니스 임팩트가 우선

### 건강한 회의주의
- 숫자가 너무 깔끔하면 의심
- 상관관계 ≠ 인과관계, 표본/생존자 편향 항상 인지

### 청중 맞춤 커뮤니케이션
- 경영진 → 결론과 액션
- 기획자 → 세부 데이터와 조건
- 개발자 → 기술적 맥락

## 2. 리포트 구조

### 4단 구조: 정의 → 발견 → 시사점 → 제언
- Executive Summary를 가장 임팩트 있는 발견으로 시작
- 우리 팀 기존 "정의→가정→근거→결론"과 호환 — "시사점" 단계 보강

### Fact/Estimate 분리
- 이미 적용 중 — 시각적 구분까지 강화 가능

### 데이터 스토리텔링 3요소
- Data(정확한 분석) + Narrative(실세계 연결) + Visual(직관적 시각화)

### 시각화 원칙
- 한 차트에 하나의 메시지
- 핵심 변곡점에 어노테이션
- 불필요한 장식 제거

### 제언의 범위 (사용자 피드백 반영)
- X: "A/B 테스트를 제안합니다" (분석가 영역 아님)
- O: "이탈률 25%로 전체 구간 중 최대입니다. 이 구간의 이탈 원인 파악이 필요합니다"
- 데이터가 보여주는 사실 + 의사결정이 필요한 지점까지만

## 3. 게임 분석 특화

### KPI 계층 구조
- North Star (1개, 게임 건강도 대표) → Core KPIs (3~5개, 일별 모니터링) → Diagnostic KPIs (다수, 원인 drill-down)

### 퍼널 분석
- 이벤트 택소노미 정리가 전제 조건
- 온보딩/모네타이제이션/진행도 퍼널 구분

### 코호트 분석
- 전체 평균 대신 코호트별 분석 기본
- 업데이트/이벤트 전후 비교로 변경 효과 측정
- FGT부터 본격 적용 예정

### 행동 기반 세그멘테이션
- 지출 기반 (무과금/소과금/고래), 플레이 패턴, 진행도, RFM
- 전체 평균 ARPU보다 세그먼트별 ARPU가 실질적 인사이트

### 맥락적 분석
- "무엇이" 아니라 "어떤 상황에서" — 대시보드 모니터링을 넘어 디자인 의사결정 지원

---

## 우리 팀 즉시 적용 항목

1. **"So What" 테스트 의무화** — 보고서 각 섹션 끝에 시사점 필수
2. **제언은 팩트 + 의사결정 지점까지만** — 솔루션 제시 금지
3. **코호트/세그먼트 분석 기본화** — FGT부터 전체 평균 대신 코호트별
