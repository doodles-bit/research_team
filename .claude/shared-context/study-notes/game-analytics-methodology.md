## 1. 이탈(Churn) 분석

### 1-1. 이탈 정의 — 게임 컨텍스트

이탈의 정의는 분석 목적에 따라 달라진다. 단일 정의를 맹신하면 안 된다.

| 유형 | 정의 | 언제 쓰나 |
|------|------|-----------|
| 비활성 이탈 (Inactivity Churn) | N일 이상 미접속 | 리텐션 모니터링, 재활성 캠페인 대상 추출 |
| 결제 이탈 (Paying Churn) | 최근 N일 내 결제 없음 (유료 게임: 재구매/DLC 기준) | 매출 예측, LTV 모델 |
| 진행 이탈 (Progression Churn) | 특정 콘텐츠 지점 이후 진행 없음 | 콘텐츠 설계 병목 파악 |
| 감성 이탈 (Sentiment Churn) | 플레이는 하지만 리뷰/평점 하락 | 브랜드 관리, 리뷰 모니터링 |

**Hi-Fi Rush 적용:** 현재 데이터는 stage_completion(진행도) 기반이므로 "진행 이탈" 분석이 주. 비활성/결제 이탈은 데이터 부재로 분석 불가.

**KRAFTON 공식 기준 (glossary 참조):** 모바일 게임 = 연속 14일 이상 미접속. 패키지 게임(Hi-Fi Rush)에는 직접 적용 불가 — 별도 정의 필요.

### 1-2. 이탈 세그먼트 프로파일링

이탈 타이밍 기반 3개 세그먼트가 업계 표준:

| 세그먼트 | 정의 | 특성 | 대응 방향 |
|----------|------|------|-----------|
| Day 1 이탈자 | 최초 플레이 후 재접속 없음 | 온보딩 경험 불만족, 장르 미스매치 | 튜토리얼 개선, 진입 장벽 낮추기 |
| 조기 이탈자 (Early Churners) | D1~D7 이탈 | 초기 훅(hook) 부재, 초반 콘텐츠 포화 | 초반 보상 루프 강화 |
| 후기 이탈자 (Late Churners) | D7~D30 이탈 | 콘텐츠 소진, 진행 장벽, 반복성 피로 | 신규 콘텐츠, 소셜 기능, 난이도 조절 |

**참고 출처:** GameAnalytics Benchmarks Report (공개 연례 보고서 시리즈); Mixpanel Game Analytics Guide

**패키지 게임(싱글플레이) 특이점:**
- 재접속 루프가 없으므로 "진행 완주율"이 리텐션 대리 지표가 됨
- 이탈 = "진행 포기" 개념으로 재정의 필요
- Hi-Fi Rush에서 실제 적용: T01→T02 이탈 = 첫 Track 완료 후 게임 포기

### 1-3. 이탈 예측 모델링

**목적:** 이탈 가능성이 높은 유저를 사전 식별하여 리인게이지먼트 조치 취함.

**주요 접근법:**

**1) RFM 기반 단순 분류 (SQL 가능)**
- Recency(최근 접속), Frequency(접속 빈도), Monetary(결제 금액) 세 축으로 유저 점수화
- 복잡한 ML 없이 SQL만으로 구현 가능 → 팀 즉시 적용 가능
- 한계: 유저 행동의 시계열 패턴 미반영

**2) 생존 분석 (Survival Analysis)**
- Kaplan-Meier 곡선: 시간 경과에 따른 "생존(=잔류)" 확률 시각화
- Cox Proportional Hazards 모델: 어떤 행동 변수가 이탈 시점에 영향을 주는지 계수 추정
- 게임 분석에서는 "D30까지 잔류할 확률"을 시각화할 때 유용
- 검열(censoring) 개념 중요: 관측 기간 내 이탈하지 않은 유저는 censored 처리

**3) 이진 분류 모델 (ML)**
- Logistic Regression, XGBoost, Random Forest 등
- 피처: 세션 길이, 접속 빈도, 진행도, 소셜 활동, 결제 이력
- 출력: 이탈 확률 스코어 → 상위 20%를 고위험군으로 분류
- 팀 현황 대비 난이도: Databricks ML 환경 구축 필요

**흔한 함정:**
- 이탈 라벨 시점 기준이 불분명하면 모델 자체가 오염됨 (라벨 정의부터 정확히)
- 클래스 불균형(이탈 < 잔류) → SMOTE 또는 class_weight 조정 필수
- 훈련/테스트 분할을 시간 기준으로 해야 미래 데이터 leak 방지 (랜덤 분할 금지)

## 2. 퍼널(Funnel) 분석

### 2-1. 퍼널 유형 구분

| 유형 | 정의 | 게임 적용 예 |
|------|------|-------------|
| 선형 퍼널 | 단계가 고정된 순서로 진행 | 튜토리얼 단계별 완료율 |
| 비선형 퍼널 | 유저가 임의 순서로 콘텐츠 소비 | 오픈월드 탐색, 사이드퀘스트 |
| 전환 퍼널 | 무과금 → 소과금 → 고과금 경로 | F2P 수익화 |
| 진행 퍼널 | 콘텐츠 클리어/완주 경로 | Hi-Fi Rush 스테이지 완주율 (현재 적용 중) |

### 2-2. 병목 파악 방법론

**단계별 이탈률 계산:**
```
이탈률(n→n+1) = 1 - (Step n+1 도달 유저수 / Step n 도달 유저수)
```

**업계 표준 병목 판단 기준:**
- 단일 단계 이탈률 > 50%: 심각 (즉각 조사 필요)
- 단일 단계 이탈률 20~50%: 주의 (패턴 모니터링)
- 단일 단계 이탈률 < 20%: 정상 범위

**참고 출처:** Amplitude Funnel Analysis Guide (docs.amplitude.com); GameAnalytics Funnel Best Practices

**퍼널 분석 심화: 시간 기반 퍼널**
- 단순 완료율이 아닌, 각 단계 소요 시간(time-to-complete) 분석 추가
- 소요 시간이 비정상적으로 긴 구간 = 유저가 막힌 구간 (이탈 전 경고 신호)
- Hi-Fi Rush: 현재 데이터에 시간 정보 없어 미적용 → 로그 설계 제안 항목

**세그먼트별 퍼널 비교 (필수):**
- 전체 퍼널만 보면 세그먼트 간 차이가 평균에 묻힘
- 플랫폼별 / 난이도별 / 국가별 퍼널을 각각 그려야 실질적 인사이트 도출 가능
- Hi-Fi Rush: 이미 적용 중 (Xbox 포함/제외 비교)

**흔한 함정:**
- 퍼널 단계 순서를 서버 이벤트 수신 시각 기준으로 할 경우 네트워크 지연으로 순서 오염 가능 → 클라이언트 타임스탬프 기준 사용 권장
- 일정 기간(윈도우) 내 완료한 경우만 퍼널에 포함해야 의미있는 비교 가능 (무제한 기간 집계 시 장기 유저가 과대 반영됨)

## 3. 코호트(Cohort) 분석

### 3-1. 코호트 유형

| 유형 | 코호트 기준 | 게임 적용 |
|------|-------------|-----------|
| 획득 코호트 (Acquisition Cohort) | 최초 설치/가입 시점 | 월별 신규 유저의 D30 리텐션 추세 |
| 행동 코호트 (Behavioral Cohort) | 특정 행동 수행 여부 | "T03 클리어 유저"의 이후 진행율 |
| 업데이트 코호트 | 특정 패치/이벤트 이후 가입 유저 | 패치 효과 측정 |
| 세그먼트 코호트 | 플레이스타일/난이도 등 속성 | Normal vs Master 진행 패턴 비교 |

**업계에서 가장 많이 쓰는 코호트:** 획득 코호트 + D1/D7/D30 리텐션 곡선

### 3-2. 리텐션 곡선 해석

**리텐션 곡선의 3개 구간:**

1. **초기 감소 구간 (D1~D7):** 가파른 하락. 온보딩 경험과 강하게 연결됨.
2. **중기 안정 구간 (D7~D30):** 완만한 감소. 코어 루프 만족도 반영.
3. **장기 평탄 구간 (D30+):** 수평에 가까운 곡선. 진정한 고관여 유저(hardcore player) 비율.

**리텐션 벤치마크 (패키지/싱글플레이 장르 기준은 F2P와 다름):**
- F2P 모바일 기준: D1=25~40%, D7=10~15%, D30=4~6% (GameAnalytics 2024 Report)
- 패키지 게임: 접속 복귀 개념보다 "완주율"이 더 의미있는 지표

**중요: 리텐션 곡선 형태 해석**
- J-커브(초기 하락 후 반등): 이벤트/업데이트 효과 or 온보딩 장벽 통과 후 만족
- L-커브(급락 후 평탄): 코어 유저 분리 빠름 — Hi-Fi Rush 현재 패턴과 유사
- 지속 하락: 코어 루프 자체에 문제 있을 가능성

### 3-3. 코호트 분석 실전 가이드

**접속일수 기반 리텐션 계산 (팀 현황):**
```sql
-- 획득 코호트별 D7 리텐션 예시
SELECT
  DATE_TRUNC('month', first_play_date) AS cohort_month,
  COUNT(DISTINCT user_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN days_since_first_play <= 7 AND last_play_date >= first_play_date + 7 THEN user_id END) AS retained_d7,
  retained_d7 / cohort_size AS retention_d7
FROM user_play_table
GROUP BY cohort_month
```

**흔한 함정:**
- 코호트 크기가 너무 작으면 노이즈가 많아 추세 해석 불가 → 주(week) 단위 코호트로 묶기
- "오늘 기준" 최신 코호트는 관측 기간이 짧아 D30 리텐션 산출 불가 → 시각화 시 미완성 코호트 표시 필수
- 플랫폼 업데이트/서버 점검 일자가 특정 코호트 리텐션을 왜곡할 수 있음 → 이상치 코호트 별도 확인

## 4. 유저 세그멘테이션

### 4-1. 세그멘테이션 접근법

| 방법 | 장점 | 단점 | 팀 적용 가능성 |
|------|------|------|----------------|
| 규칙 기반 (Rule-based) | SQL로 즉시 구현, 해석 쉬움 | 주관적 기준, 경계 불명확 | 즉시 적용 가능 |
| RFM 스코어링 | 체계적, 업계 표준 | 게임 행동 다양성 미반영 | 즉시 적용 가능 |
| K-means 클러스터링 | 데이터 기반, 자동 그룹화 | 해석 어려움, k 선택 주관적 | Databricks 환경에서 가능 |
| 잠재 클래스 분석 (LCA) | 범주형 변수에 강함 | 통계 전문성 필요 | 중장기 |

### 4-2. 행동 기반 세그멘테이션 실전

**게임에서 자주 쓰는 피처 (행동 변수):**
- 세션 길이 / 세션 빈도
- 특정 콘텐츠 클리어 여부
- 사용 스킬/캐릭터 구성 (Hi-Fi Rush: 파트너 vs 콤보 비율)
- 결제 이력 / 결제 금액
- 소셜 행동 (PvP, 길드, 채팅)

**Hi-Fi Rush 현재 적용 (Track 1):**
- 공격형 / 방어형 / 균형형 3분류는 규칙 기반 세그멘테이션
- 스테이지별 스킬 사용 비율을 기준으로 분류 → 업계 표준 방식과 일치

**심화: 시간 기반 세그멘테이션 (진화형 세그먼트)**
- 유저 세그먼트는 고정이 아니라 시간에 따라 변함
- "Week 1 공격형 → Week 4 균형형"처럼 세그먼트 이동 추적이 가능하면 인사이트 깊어짐
- Hi-Fi Rush: 스테이지 진행에 따른 플레이스타일 전환 분석 = 이미 적용 중

**흔한 함정:**
- 너무 많은 세그먼트: 10개 이상이면 실무 활용 불가 → 3~5개가 적정
- 세그먼트 간 크기 불균형: 한 세그먼트에 90% 몰리면 의미없음 → 크기 분포 항상 확인
- 세그먼트에 이름 붙이기의 함정: "고래"라고 부른다고 다 같은 행동 패턴이 아님 → 이름보다 행동 정의가 중요

## 5. 통제 비교 분석 (Confounding Variable Control)

### 5-1. 핵심 개념

**Confounding variable(혼재 변수):** 원인 변수와 결과 변수 모두에 영향을 미치는 제3의 변수. 통제하지 않으면 허위 상관(spurious correlation)이 발생함.

**Hi-Fi Rush 실제 사례:**
- "Master 난이도 완주율이 Normal보다 높다" → 원인이 난이도 자체가 쉬워서가 아님
- 자기선택 편향(Self-selection Bias): Master를 선택한 유저 자체가 코어 게이머 → 선발 효과

### 5-2. 주요 통제 방법

**1) 동일 코호트 비교 (Same-Cohort Comparison)**
- 같은 기간에 시작한 유저끼리만 비교
- 계절성, 마케팅 효과 등 시간 관련 혼재 변수 통제
- 팀 적용: 특정 국가의 이탈률 비교 시 "같은 달 시작 유저"끼리만 비교

**2) Day 1 비교 (Baseline Comparison)**
- 모든 유저를 D1 상태에서 동등하게 보고 이후 행동 비교
- 접속 기간이 다른 유저 간 비교 시 발생하는 편향 제거
- 팀 현황: Xbox 포함/제외 비교 시 이미 암묵적으로 적용 중

**3) 매칭 (Matching)**
- 비교 그룹 간 관찰 가능한 특성(나이, 플랫폼, 진행도)을 일치시킴
- Propensity Score Matching(PSM): 처치 그룹(예: Master 선택)과 유사한 특성의 대조 그룹 구성
- 한계: 관찰 불가능한 특성(동기, 스킬)은 통제 불가

**4) 차분-인-차분 (Difference-in-Differences, DiD)**
- 처치 전후 변화와 대조군 전후 변화를 비교
- A/B 테스트 없이도 정책/패치 효과 추정 가능
- 조건: 처치 전 두 그룹이 유사한 추세를 가져야 함(평행 추세 가정)
- 팀 적용 가능: 특정 패치 전후 이탈률 변화를 플랫폼별로 비교

**5) 회귀 분석으로 통제 변수 포함**
- 관심 변수 외에 혼재 변수를 회귀식에 포함하여 효과 분리
- 예: `이탈률 ~ 난이도 + 플랫폼 + 국가 + 접속일수` → 난이도의 순수 계수 추정

**흔한 함정:**
- "통제했으니 인과관계다"라고 단정 금지 → 관찰 데이터 기반 분석은 항상 잔존 혼재 가능성 있음
- 너무 많은 변수 통제(과통제): 인과 경로 상의 변수를 통제하면 오히려 효과가 사라짐 (콜라이더 편향)
- A/B 테스트가 가능하다면 관찰 연구보다 A/B 테스트가 우선 — 관찰 분석은 A/B 불가 시 차선책

**참고 출처:** Pearl J. (2009) Causality; Cunningham S. (2021) Causal Inference: The Mixtape (공개 교재 mixtape.scunning.com)

## 6. 통계 검정

### 6-1. 게임 분석에서 자주 쓰는 검정 목록

| 상황 | 검정 방법 | 조건 |
|------|-----------|------|
| 두 그룹 완주율 비교 | Two-proportion z-test | 각 그룹 n >= 30, 독립 표본 |
| 두 그룹 평균 세션 시간 비교 | t-test (독립 표본) | 정규성 가정 필요, 위반 시 Mann-Whitney |
| 세 그룹 이상 평균 비교 | One-way ANOVA + 사후검정 | 등분산 가정 |
| 클릭률 / 전환율 A/B 테스트 | Chi-square 또는 z-test | 기대 빈도 >= 5 |
| 시계열 이상 탐지 | 관리도(Control Chart) | 과거 데이터 충분할 때 |

**팀 현재 적용 (cross-team-insights 참조):**
- Wilson 95% CI + Two-proportion z-test 이미 도입됨 (2026-03-31)

### 6-2. p-value 해석 가이드 (흔한 오해 정리)

**p-value가 의미하는 것:**
- "귀무가설(H0)이 참일 때, 이 결과(또는 더 극단적인 결과)가 관찰될 확률"
- p < 0.05: 유의 수준 5%에서 귀무가설 기각 → "통계적으로 유의하다"

**p-value가 의미하지 않는 것:**
- "효과 크기가 크다"는 의미 아님 → 표본이 크면 아주 작은 차이도 p < 0.05
- "실무적으로 중요하다"는 의미 아님 → 비즈니스 맥락에서 효과 크기(Effect Size) 별도 평가 필수
- "귀무가설이 거짓임을 증명"하지 않음 → 기각의 근거 제공일 뿐

**게임 분석 실전 룰:**
1. 통계적 유의성(p < 0.05) + 실무적 유의미성(최소 효과 크기 사전 정의) 둘 다 충족해야 "의미있는 차이"로 서술
2. 95% CI(신뢰구간)를 함께 보고 → 구간이 넓으면 추정 불확실성 높음
3. 다중 비교(multiple comparisons) 문제: 여러 그룹을 동시에 비교하면 우연히 p < 0.05가 나올 확률 증가 → Bonferroni 보정 또는 FDR 보정 적용

### 6-3. 효과 크기 (Effect Size)

통계적 유의성만으로는 부족. 효과 크기를 반드시 보고.

| 검정 | 효과 크기 지표 | 해석 기준 |
|------|----------------|-----------|
| t-test | Cohen's d | small=0.2, medium=0.5, large=0.8 |
| 비율 비교 | 상대 위험비(RR), 오즈비(OR) | 1에서 멀수록 큰 효과 |
| ANOVA | Eta-squared (η²) | small=0.01, medium=0.06, large=0.14 |
| 카이제곱 | Cramér's V | small=0.1, medium=0.3, large=0.5 |

**팀 적용 포인트:**
- 퍼널 비교 시 이탈률 차이(절대값 + 상대값 모두 보고)
- 예: "Normal T01 완주율 16.2%, Master 33.4% — 절대 차이 17.2%p, 상대 차이 +106%"
- 이미 팀이 절대값+상대값 병기 원칙 따르고 있음 → 효과 크기 지표 추가로 강화 가능

### 6-4. A/B 테스트 설계 (향후 적용 대비)

**필수 고려 사항:**
1. 샘플 크기 사전 계산: 검정력(power) 80%, 유의수준 5%, 최소 탐지 효과(MDE) 정의 후 필요 n 계산
2. 랜덤화 단위: 유저 단위 vs 세션 단위 (유저 단위 원칙 — 동일 유저가 A/B 양쪽 경험 금지)
3. 실험 기간: 최소 1주(요일 효과 통제), 권장 2~4주
4. 조기 종료 금지: "p < 0.05 보이자마자 종료"는 1종 오류 증가 → 사전에 정한 기간 준수

**참고 출처:** Kohavi R. et al. (2020) Trustworthy Online Controlled Experiments (Cambridge Univ. Press); Optimizely Stats Engine 문서

## 우리 팀 즉시 적용 권장 항목 (상위 3건)

> 기존 `20260327-analysis-best-practices.md`의 "즉시 적용 항목"과 중복되지 않는 내용.

### 1순위: 효과 크기(Effect Size) 병기 의무화
- **현재:** 통계 검정(Wilson CI + z-test) 도입 완료 (2026-03-31)
- **추가:** 비율 비교 시 Cohen's h 또는 절대/상대 차이를 효과 크기 맥락에서 해석하는 문장 추가
- **이유:** p-value만으로는 "얼마나 큰 차이인가"를 말할 수 없음. 보고서 수신자(비즈니스팀)가 실무 판단을 하려면 효과 크기가 필요.
- **적용 난이도:** 낮음. 기존 수치에 해석 문장만 추가.

### 2순위: 이탈 세그먼트 프로파일링 도입 (진행 이탈 기반)
- **현재:** 퍼널 이탈률 계산은 완료. 이탈자 특성 분석(누가 이탈하는가) 미수행.
- **추가:** 이탈 시점(T01 vs T03 vs T06)별로 이탈자의 플레이스타일/플랫폼/국가 분포 비교
- **이유:** 이탈률 수치만으로는 "왜 이탈하는가"를 알 수 없음. 세그먼트 프로파일링으로 유형별 원인 가설 도출 가능.
- **적용 조건:** 현재 보유 데이터(stage_completion + platform + country CSV)로 구현 가능.

### 3순위: 코호트 비교 기반 퍼널 해석 강화
- **현재:** 전체 집계 기준 퍼널 분석. 코호트별 분리 미적용.
- **추가:** 플랫폼별 획득 코호트(예: Steam 유저 vs Xbox 유저)의 퍼널을 동일 기간 기준으로 비교
- **이유:** 전체 퍼널은 플랫폼 간 유저 구성 차이를 흡수. 코호트 분리로 "플랫폼 자체 효과"와 "유저 구성 효과"를 분리 가능.
- **적용 조건:** stage_completion 테이블에 platform 컬럼 존재 — 데이터 지원 가능.
