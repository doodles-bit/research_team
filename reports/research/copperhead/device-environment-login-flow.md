## 1. 요약

Copperhead 베타의 PC 하드웨어, 접속 환경, 로그인 흐름을 분석했다. **134대의 고유 기기 중 60.4%(81대)가 Parsec 등 원격/가상 접속 어댑터를 사용**하고 있었고, 물리 GPU를 장착한 기기는 59대(44%)에 불과했다. 세션 데이터를 분석한 결과, 전체 11,285명 중 **60.3%(6,805명)가 자동화/공유 테스트 머신(PFLIGHT, MINSPEC, RECSPEC)에서 접속**했으며, 이 머신들은 기기당 800~1,395명의 계정을 운용하는 대규모 자동 테스트 환경이었다. 로그인 성공률은 98.7%로 매우 높았고, 실패 104건은 모두 "HttpResponse is null" 오류였다.

## 2. 연구 배경

이전 4건의 연구에서 반복적으로 나타난 의문이 있었다:

- 1차 연구: 미션 텔레메트리 누락률이 80.7%에 달하는데, 원인이 자동 테스트인가 실제 이탈인가?
- 2차 연구: heartbeat에서 SDS- 기기 33대, BBI- 5대가 관찰되었는데, 전체 테스트 환경 구조는?
- 3차 연구: 세션 참여 실패율 12.1%가 특정 환경에서 집중되는 것은 아닌가?

아직 탐색하지 않은 `view_client_gpp_device_info`(PC 하드웨어 정보), `view_client_gpp_user_entry`(로그인 흐름), `sessionstart`/`sessionend`(텔레메트리 세션) 테이블을 분석해, 베타 참여 환경의 전체 구조를 파악하고 데이터 해석에 필수적인 맥락을 제공하고자 했다.

## 3. 가설

### H1: 베타 접속 기기의 절반 이상이 원격 접속(Parsec 등) 환경이다

- **예상 결과**: 원격 접속 어댑터 GPU 비율 > 50%
- **기각 조건**: 원격 접속 비율 < 30%이면 기각
- **근거**: 2차 연구에서 SDS 기기에 Parsec 등 원격 접속 흔적이 관찰됨

### H2: 자동화 테스트 머신(PFLIGHT, MINSPEC, RECSPEC)이 전체 세션의 상당 비율을 차지한다

- **예상 결과**: 자동화 머신의 세션 비율 > 40%
- **기각 조건**: 자동화 머신 비율 < 20%이면 기각
- **근거**: SDS-PFLIGHT-05에서 다수 계정 동시 접속 패턴이 3차 연구에서 관찰됨

### H3: 로그인 실패(login_failed)가 특정 기기 또는 환경에 집중되어 있다

- **예상 결과**: 로그인 실패의 50% 이상이 상위 5개 기기에서 발생
- **기각 조건**: 기기 간 실패율이 균등하면(상위 5개 기기 비율 < 30%) 기각
- **근거**: 3차 연구에서 세션 참여 실패가 76명에 집중된 패턴

상위 5개 기기가 전체 104건 중 78건(75.0%)을 차지했다. 그러나 이 기기들은 모두 전체 세션 수가 500건 이상으로 대규모 자동 테스트 기기이며, 기기별 실패율은 0.6~2.5%로 매우 낮다. 실패는 **2월~4월 전 기간에 걸쳐 산발적으로 발생**했고, 특정 날짜에 급증하는 패턴은 없었다.

#### 날짜별 실패 건수 추이

33개 날짜에 걸쳐 1~11건씩 분산 발생. 최대일은 2/25(11건)과 4/8(9건). 특정 날짜에 집중적으로 실패가 발생하는 "장애" 패턴은 관찰되지 않았다.

## 5. 반증 탐색 결과

### 5.1 "원격 접속 기기가 많다 = 원격 플레이"라는 해석의 한계

GPU가 Parsec Virtual Display Adapter로 보고되더라도, 이것이 반드시 원격 접속 중이었다는 뜻은 아니다. Parsec이 설치된 기기라면 로컬 사용 중에도 해당 어댑터가 보고될 수 있다. 다만, 9대의 기기(6.7%)가 시기에 따라 원격 GPU와 물리 GPU 사이를 전환한 것으로 기록되어, 적어도 일부는 실제로 원격/로컬을 번갈아 사용한 것으로 확인된다.

### 5.2 자동화 머신의 미션 참여율은 높다

자동화 테스트 머신(PFLIGHT+MINSPEC+RECSPEC)의 계정 중 29.1%(1,983/6,805)가 미션 데이터에도 등장했다. 이는 비자동화 머신(23.3%, 1,042/4,480)보다 오히려 높다. 특히 RECSPEC(51.1%)과 MINSPEC(48.6%)은 미션 참여율이 높아, 이 머신들이 "로그인만 하고 나가는" 것이 아니라 **실제 미션까지 수행하는 자동 테스트**를 돌리고 있음을 보여준다. PFLIGHT는 14.7%로 상대적으로 낮아, 로그인/세션 생성 위주 테스트로 보인다.

| 머신 유형 | 전체 계정 | 미션 참여 계정 | 미션 참여율 |
|----------|---------|------------|----------|
| PFLIGHT | 4,023 | -- | 14.7% (세션 기준) |
| RECSPEC | 1,395 | -- | 51.1% (세션 기준) |
| MINSPEC | 1,388 | -- | 48.6% (세션 기준) |
| 비자동화 전체 | 4,480 | 1,042 | 23.3% |

> 출처: `sessionstart` + `cphmissionstarted`, account_id 조인

이 발견은 **1차 연구의 미션 텔레메트리 누락(80.7%)의 부분적 원인을 설명**한다: MINSPEC/RECSPEC에서 미션을 시작하지만, 자동 테스트이므로 정상적인 성공/실패 이벤트 없이 프로세스가 종료될 수 있다.

### 5.3 로그인 실패 집중은 단순히 사용량에 비례

상위 5개 실패 기기가 75%의 실패를 발생시켰지만, 이 기기들은 동시에 전체 세션의 높은 비중을 차지하는 대규모 사용 기기였다. 기기별 실패율(0.6~2.5%)에서는 뚜렷한 환경 의존적 패턴이 보이지 않았다.

## 6. 결론 및 시사점

### 6.1 베타 데이터의 60%는 자동화 테스트에서 생성된다

전체 11,285 계정 중 6,805명(60.3%)이 7대의 자동화/공유 테스트 머신에서 접속했다. 이 계정들은 기기당 800~1,395명 규모로 생성되며, 각 계정은 단 1대의 기기에서만 사용된다. **베타 데이터 분석 시 자동화 테스트 계정을 식별하고 필요에 따라 분리해야 정확한 "실제 사용자" 행동을 파악할 수 있다.**

- **식별 기준 제안**: `computername`이 `*PFLIGHT*`, `*MINSPEC*`, `*RECSPEC*`에 해당하는 세션
- 단, 이 기준은 `sessionstart`에서만 적용 가능하며, `cph*` 이벤트 테이블에는 computername이 없는 경우도 있음

### 6.2 개발 환경의 60%가 원격 접속 기반

134대 기기 중 81대(60.4%)가 Parsec 등 원격/가상 접속 어댑터를 사용했다. 이는 게임의 성능 관련 텔레메트리 해석에 주의가 필요한 지점이다:

- 2차 연구에서 발견된 FPS 이상치(초고 FPS, 초저 FPS)의 일부가 원격 접속 환경에서 발생했을 가능성이 있다
- Parsec 등은 렌더링을 원격 서버에서 수행하므로, 클라이언트 측 FPS 수치가 실제 렌더링 성능을 반영하지 않을 수 있다

### 6.3 로그인 품질은 양호, 서버 연결 에러만 주의

로그인 성공률 98.7%는 베타 테스트 환경에서 매우 양호한 수준이다. 104건의 실패는 모두 "HttpResponse is null"(서버 응답 없음) 에러로, 네트워크 일시 단절이나 서버 불안정이 원인으로 보인다. 특정 환경에 국한된 문제라기보다 산발적 네트워크 이슈다.

### 6.4 세션 종료 텔레메트리 누락은 기기 유형에 따라 극단적 차이

| 환경 | 세션 종료 누락률 |
|------|--------------|
| PFLIGHT (동시 다중 계정 서버) | 85.4% |
| MINSPEC/RECSPEC (자동 테스트) | 50~53% |
| SDS Dev / BBI / External | 11~13% |

자동화 테스트 머신의 세션 종료 누락률이 50~85%에 달한다는 것은, 전체 세션 데이터에서 이 기기들을 분리하지 않으면 "세션 이탈률"이 과대 추정될 수 있음을 의미한다.

## 7. 한계 및 후속 연구

### 7.1 데이터 한계

- **device_info와 sessionstart 간 직접 조인 불가**: `view_client_gpp_device_info`는 `device_id`/`analytics_id`를, `sessionstart`는 `account_id`/`computername`을 사용한다. 두 테이블 간 정확한 매핑이 어려워, GPU 등급별 FPS/세션 행동 교차 분석은 수행하지 못했다.
- **computername 기반 분류의 한계**: 기기 이름이 변경되거나 비표준적인 경우 분류가 부정확할 수 있다. "External/Other"로 분류된 61대 중 일부는 SDS 내부 기기일 가능성이 있다.
- **Parsec 어댑터 = 원격 접속 확정 아님**: 앞서 언급한 대로, Parsec이 설치된 기기라면 로컬 사용 중에도 해당 GPU가 보고될 수 있다.

### 7.2 후속 연구 제안

1. **자동화 테스트 제외 후 미션 분석 재검증**: 1차 연구의 미션 누락률 80.7%가 자동 테스트 필터링 후 어떻게 변하는지 확인
2. **device_info와 heartbeat 연계**: `device_id` 또는 `analytics_id`를 매개로 GPU 등급과 FPS 데이터를 연결하여, 하드웨어 사양별 성능 차이 분석
3. **자동 테스트 빌드 버전 패턴 분석**: 자동화 머신이 952개 고유 빌드를 테스트했다는 점에서, 빌드 간 성능/안정성 변화 추적이 가능할 수 있음
