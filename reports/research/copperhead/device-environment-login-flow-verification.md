## 검증 요약

리포트의 핵심 수치 9개 항목을 독립 쿼리로 검증한 결과, **1건의 중대 오류(GPU 소계 불일치)**와 **다수의 경미한 수치 차이(1~3건 수준)**를 발견했다. 가설 판정 논리는 대체로 타당하나, H1의 근거 수치인 "원격/가상 56%(75대)"는 소계 오류로 인해 재산정이 필요하다. 확증 편향은 제한적이며, 반증 탐색은 양호했다.

**종합 판정: MINOR** -- GPU 소계 오류를 수정하고, 경미한 수치 차이를 반영하면 진행 가능.

### a) GPU 유형별 기기 분포: [중대 오류 발견]

**리포트 수치**: 134대 중 Parsec 75, MS Remote 4, Virtual Monitor 1, DisplayLink 1 → **소계: 원격/가상 = 75 (56.0%)**

**검증 결과**:
| GPU 유형 | 리포트 | 검증 | 일치 |
|----------|--------|------|------|
| Parsec | 75 | 75 | O |
| MS Remote Desktop | 4 | 4 | O |
| Virtual Monitor | 1 | 1 | O (주1) |
| DisplayLink | 1 | 1 | O |
| 전체 고유 기기 | 134 | 134 | O |
| **소계: 원격/가상** | **75 (56.0%)** | **81 (60.4%)** | **X -- 중대 오류** |

> 출처: `view_client_gpp_device_info`, `event_date >= '2026-02-04'`, `COUNT(DISTINCT device_id)` 기준

**오류 내용**: 리포트의 개별 행(Parsec 75 + MS Remote 4 + Virtual Monitor 1 + DisplayLink 1)의 합은 81이지만, 소계 행은 75(=Parsec만)로 표기되어 있다. 소계가 Parsec만을 반영한 것으로, MS Remote(4) + Virtual Monitor(1) + DisplayLink(1) = 6대가 누락되었다.

**추가 발견**: 원격/가상 GPU를 한 번이라도 보고한 기기 = 81대이지만, 이 중 9대는 시기에 따라 물리 GPU도 보고했다(리포트 5.1절에서 정확히 9대로 언급). 따라서:
- 원격/가상 GPU 기록이 있는 기기 = **81대 (60.4%)**
- 순수 원격 전용 기기 = **72대 (53.7%)**
- 물리 GPU 전용 기기 = **53대 (39.6%)**
- 양쪽 모두 = **9대 (6.7%)**

**H1 판정에의 영향**: 어떤 기준을 적용하든(81대 또는 72대) 가설의 기각 조건(30% 미만)을 충족하지 않으므로 H1 채택은 유지된다. 단, 본문의 "56%" 수치는 "60.4%(81대)" 또는 "53.7%(72대, 순수 원격 전용)"로 수정해야 한다.

(주1) 리포트는 "Meta Virtual Monitor"가 아닌 "VirtualMonitor Device"로 표기. 실제 데이터에는 "Meta Virtual Monitor" 1대, "VirtualMonitor Device" 1대로 2건이 있으나, 첫 번째 GPU 기준 분류 시 중복 제거 후 각 1대로 카운트됨.

### b) 자동화 환경 분류 수치: [경미한 차이]

**리포트 수치**: PFLIGHT 4,023명, MINSPEC 1,387명, RECSPEC 1,394명, 합계 6,805명/11,284명 = 60.3%

| 항목 | 리포트 | 검증 | 차이 |
|------|--------|------|------|
| PFLIGHT 계정 | 4,023 | 4,023 | 일치 |
| MINSPEC 계정 | **1,387** | **1,388** | **+1** |

> 출처: `sessionstart`, `event_date >= '2026-02-04'`, `COUNT(DISTINCT account_id)` 기준

(주2) 환경별 개별 DISTINCT 합(4,023+1,388+1,395 = 6,806)과 전체 DISTINCT(6,805)에 1건 차이 존재. 이는 1개 계정이 두 환경(예: RECSPEC과 다른 환경)에 걸쳐 있음을 의미. 리포트의 "모든 계정이 정확히 1대의 computername에서만 세션을 시작" 주장은 0개의 복수 머신 사용 계정으로 확인되어 맞지만, 환경 분류의 LIKE 패턴이 겹치는 경우가 1건 있을 수 있다.

**전체 계정 합계 검증**: 4,023 + 1,388 + 1,395 + 2,411 + 1,449 + 620 = 11,286 (개별 DISTINCT 합). 전체 DISTINCT = 11,285. 1건 중복은 환경 간 계정 겹침으로, 리포트의 "11,284" 표기와 1건 차이.

### c) 세션 종료 누락률: [SDS Dev, External, BBI 차이 발견]

**검증 결과** (DISTINCT sessionid 기준):
| 환경 | 리포트 시작 | 검증 시작 | 리포트 누락률 | 검증 누락률 | 비고 |
|------|-----------|---------|------------|----------|------|
| PFLIGHT | 4,038 | 4,035 | 85.4% | 85.4% | 일치 |
| MINSPEC | 1,388 | 1,388 | 53.1% | **53.2%** | 0.1%p 차이 |
| RECSPEC | 1,395 | 1,395 | 50.6% | 50.6% | 일치 |
| SDS Dev | **2,771** | **2,428** | **11.3%** | **12.9%** | **차이 있음** |
| BBI | **676** | **622** | **11.4%** | **11.7%** | **차이 있음** |
| External | **1,525** | **1,450** | **13.0%** | **13.7%** | **차이 있음** |

> 출처: `sessionstart LEFT JOIN sessionend ON sessionid`, `event_date >= '2026-02-04'`, `COUNT(DISTINCT sessionid)` 기준

**원인**: sessionstart 테이블에 동일 sessionid의 중복 행이 존재한다. 리포트는 COUNT(*)를 사용했고, 검증은 COUNT(DISTINCT sessionid)를 사용했다. SDS Dev에서 특히 큰 차이(343건, 2,771 vs 2,428)가 발생한다. 어느 방법이 맞는지는 중복 행의 성격(실제 중복 이벤트인지, 데이터 수집 중복인지)에 따라 다르나, 세션 누락률을 정확히 계산하려면 DISTINCT sessionid 기준이 더 적절하다.

PFLIGHT/MINSPEC/RECSPEC의 누락률은 차이가 거의 없어 리포트의 핵심 메시지("자동화 머신의 세션 종료 누락률이 50~85%")에 영향 없다. 그러나 **SDS Dev의 누락률은 11.3%가 아닌 12.9%**, External은 13.0%가 아닌 13.7%로 수정이 필요하다.

## 논리 검토

### H1: 원격 접속 기기 비율 > 50% → 채택

- **소계 오류 영향**: 소계를 81로 수정하면 60.4%(기각 조건 30% 훨씬 초과)로 채택이 더 강화됨
- 순수 원격 전용(72대, 53.7%)로 보더라도 채택 유지
- **논리 평가**: 타당. 단, 리포트가 "56%"를 여러 곳에서 반복 사용하므로 모두 수정 필요

### H2: 자동화 머신 세션 비율 > 40% → 채택

- 60.3% = 기각 조건 20%를 크게 초과
- PFLIGHT 4,023명, MINSPEC 1,388명(리포트 1,387과 1건 차이), RECSPEC 1,395명(리포트 1,394와 1건 차이)
- **논리 평가**: 타당. 수치 차이는 결론에 영향 없음

### H3: 로그인 실패 집중도 → 부분 채택

- 상위 5기기 = 75.0% (기각 조건 30% 초과) → 집중도 측면에서 채택
- 그러나 기기별 실패율 0.6~2.5%로 매우 낮음 → "실무적 의미 제한적"이라는 해석은 적절
- **논리 평가**: 타당. "부분 채택"은 집중도(존재) + 실무 영향(미미)을 균형 있게 반영

## 확증 편향 검토

### 반증 탐색 평가

리포트는 4건의 반증을 제시했다:

1. **5.1 Parsec 설치 = 원격 접속 아님**: 적절한 반증. Parsec이 설치만 되어도 GPU로 보고될 수 있다는 한계를 명시.
2. **5.2 자동화 머신의 미션 참여율이 높다**: "자동화 = 로그인만 테스트"라는 예상에 반하는 증거를 제시. RECSPEC 51.2%, MINSPEC 48.6%로 오히려 비자동화(23.3%)보다 높음. 적절한 반증.
3. **5.3 실패 집중 = 사용량 비례**: 적절한 반증. 실패가 특정 환경 때문이 아니라 사용량에 비례한다는 점을 지적.
4. **5.4 모든 계정이 1대 기기**: 반증이라기보다 "자동 테스트 일회용 계정" 패턴의 추가 근거. 반증으로 분류하기 어려움.

**평가**: 4건 중 3건이 실질적 반증이며, 특히 5.2(미션 참여율)은 기존 가정을 뒤집는 좋은 발견. 그러나 **5.4는 반증이 아닌 보강 근거**로, 이를 반증 섹션에 둔 것은 부적절하다. 실질적 반증 3건은 이 규모의 탐색적 연구에 충분한 수준이다.

### 확증 편향 징후 검토

- "자동화 테스트가 베타 데이터의 상당 부분을 차지한다"는 결론은 데이터에 충분히 뒷받침됨
  - 기기당 800~1,394명의 계정 운용, 세션 중앙값 17.6분(좁은 분포), 세션 종료 누락 50~85%
  - 이 패턴들이 일관되게 "자동화 테스트"를 가리킴
- **잠재적 편향**: "원격 접속 = Parsec이 설치된 기기 = 개발/QA 환경"이라는 해석은 맞을 가능성이 높으나, 데이터만으로 100% 확정하기는 어려움. 리포트가 5.1절에서 이 한계를 명시한 것은 적절.
- **누락된 반증 탐색**: "External 61대 중 실제로 외부 테스터가 아닌 기기가 있을 수 있다"는 가능성에 대한 탐색이 없음. 7.1절에서 한계로 언급했으나 반증 차원의 탐색은 미실시.

| 우선순위 | 항목 | 내용 |
